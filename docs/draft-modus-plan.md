# Draft-Modus Design Plan

**Audience:** an implementing agent working in this repo.
**Status:** Phases 1–4 and 6 shipped on `main`, including a Phase 4 follow-up
(manual-trigger rule bypass, on-demand generation, local Ollama auto-start,
a newsletter-address hard gate) verified live in Mail.app (see their entries
below). Phase 5 remains a roadmap stub.

Referenced by `docs/draft-contract.md` and `docs/style-learning-contract.md`
as "the Draft-Modus design plan" — this file is that plan. It covers
scheduling, newsletter filtering, and macOS integration; the prompt-assembly
details those two contract docs govern are out of scope here.

Draft-Modus is opt-in (`SettingsStore.draftModeEnabled`, default off), never
touches Apple Mail's real Drafts folder, and stores everything in a separate
local database (`drafts.db`, decoupled from `askmail.db`) so deleting or
rebuilding one never touches the other.

---

## What's shipped

### Phase 1 — drafting core (`7d97d8f`)

Thread reconstruction (`ThreadResolver`, via `In-Reply-To`/`References`),
newsletter/bulk-mail classification (`NewsletterClassifier`), a shared
hybrid-retrieval helper extracted from `QueryService` (`Retriever`), prompt
assembly (`DraftAssembler`, `docs/draft-contract.md`), and the separate
`drafts.db` (`DraftStore`). Fully unit-testable, zero new OS permissions, no
background scheduling — this phase never runs anything on its own.

### Phase 2 — background orchestration (`a6be7d3`)

`DraftEngine`/`DraftScheduler` (`Sources/AskMailApp`) + `DraftJobProcessor`
(the testable core, `Sources/AskMailCore`): a fast envelope-index poll for new
Inbox mail, an FSEvents watch on `INBOX.mbox` with a 2-minute floor `Timer` as
a safety net, wake-from-sleep and launch catch-up, and a job queue
(`draft_jobs`) carrying each candidate through
`pending -> classifying -> {newsletterSkipped | autoGenerated | eligible} ->
drafting -> {drafted | failed, backoff-gated retry}`.

### Phase 3 — style learning (`2b71b86`, `aadb615`)

`StyleLearner` (`docs/style-learning-contract.md`): compares a generated
draft against the real Sent reply that eventually followed it, and folds the
delta into `global`/`domain`/`address`-scoped `style_profiles` rows via a
local-only LLM merge call, self-gated to once/24h. `DraftJobProcessor`
now looks up the best-available learned guidance (address > domain > global)
before assembling each draft — the `styleGuidance` hook Phase 1 built but
always passed `nil`.

While reviewing Phase 3, a **pre-existing Phase 2 gap** was also found and
fixed (`aadb615`, not really "Phase 3" work): `NewsletterClassifier`'s own
doc comment claimed a caller-side `noreply@`/`no-reply@` hard rule already
existed; it didn't. `NewsletterClassifier.isNoReplySender` now closes that,
checked the same way as the existing `isAutoGenerated` gate.

### Phase 4 — `DraftService`: macOS Services menu (shipped)

`DraftServiceProvider` (`Sources/AskMailApp`), registered via
`NSApplication.servicesProvider` + `Packaging/Info.plist`'s `NSServices`
array: two Mail ▸ Services items, "AskMail: Insert Draft" and "AskMail:
Regenerate Draft" — standard, unprivileged macOS mechanism, no entitlement
change, no FDA/Automation/Accessibility prompt.

The plan's own open question — "what does a Services invocation actually
hand the provider" — was resolved with a **live-Mac spike** before writing
the provider, exactly as planned: built a diagnostic-only `NSServices`
provider, registered it, and invoked it against a real Mail.app compose
window. Finding: Mail hands the provider only the current pasteboard
selection (plain text + RTF) — never a message id or any AppleScript-level
identifier, confirming the plan's own prediction. The selection is the
auto-inserted reply quote, whose first line is Mail's standard `"On <date>,
<Name> <<email>> wrote:"` header followed by the quoted original body
verbatim — the only identifying signal available without Phase 5's
Automation grant.

`DraftServiceMatcher` (`Sources/AskMailCore`, fully unit-tested) is the
mechanism this settled on: extracts the quoted sender's email from the
selection, filters `DraftStore`'s `ready` drafts to that sender, and — when
the same correspondent has more than one open thread — disambiguates by
checking which candidate's original message body (via `askmail.db`) has the
longest normalized-text overlap with the selection, falling back to
most-recently-generated when that can't resolve it. "Insert" therefore
**requires the user to select the quoted text first** and replaces that
selection with the draft — not a bare insert-at-cursor leaving the quote
untouched, since an empty selection carries no identifying information at
all. This is the concrete answer to the plan's "how much of Insert is even
feasible without Phase 5's Automation grant" question: fully feasible, with
that one selection-first UX requirement.

"Regenerate" bypasses the job queue and scheduler cadence entirely via a new
`DraftJobProcessor.regenerateDraft(threadID:...)` entry point that reuses
`draftOne`'s exact retrieval/style/assembly path (refactored into a shared
`generateDraftText` helper) rather than duplicating it. It's genuinely
asynchronous (a local-LLM call) and declares no `NSReturnTypes`, so it
returns immediately after kicking off a detached task and notifies via
`DraftNotifier` on completion — the next "Insert" then reflects the fresh
draft.

Phase 6's sender-exclusion list (below) is enforced the same way regardless
of which surface triggered drafting — it's a `classifyPendingJobs` gate, not
something Insert/Regenerate need to know about separately.

A 5-dimension adversarial review (concurrency, hardening/privacy,
state-machine correctness, Services-menu-specific edge cases, integration/
regression) of the Phase 4 + Phase 6 diff, per the process note above, found
and every finding was independently re-verified before being trusted. Nine
real, non-obvious issues survived verification and were fixed:

- **Most severe:** `DraftServiceMatcher`'s candidate filter had a
  `localizedCaseInsensitiveContains` fallback that matched
  `"smith@corp.com"` against a stored `"jblacksmith@corp.com"` draft (a
  substring, not an equal address) — silently inserting/regenerating the
  *wrong correspondent's* draft. Removed; only exact address equality
  matches now.
- `quotedSenderEmail` accepted any email-shaped substring anywhere in the
  first 500 characters. Since `NSSendTypes` are generic plain-text/RTF
  types, both services are technically invocable from a text selection made
  in *any* app, not just Mail — tightened to require the address be
  angle-bracket-wrapped on the selection's first line, matching Mail's
  actual quote-header shape exactly (substantially narrows, though can't
  fully eliminate, cross-app false positives).
- `regenerateDraft` never checked `draftModeEnabled` or the sender
  exclusion list — the master opt-in toggle and "never draft for" list
  could be silently bypassed via the Services menu. Both are now checked
  (Insert remains exempt: it only reads an already-generated draft, same as
  the always-available Drafts window).
- `regenerateDraft` always `INSERT`ed a new row instead of replacing the
  thread's existing `ready` draft (Phase 1's `latestDraft` doc comment had
  flagged this as "a later phase's concern" — the diff's own doc comments
  claimed "replaces" without actually doing so). Now deletes the thread's
  prior ready draft(s) after generation succeeds (never before, so a failed
  regenerate can't destroy the existing draft).
- A Services "Regenerate" fired its own notification *and* got picked up
  by the next scheduled tick's cursor-based notifier, double-notifying for
  one action. `DraftEngine.skipNotifying` now advances that cursor past the
  regenerated row first.
- A large/adversarial pasteboard selection made `insertDraft`'s (necessarily
  synchronous) disambiguation pass do unbounded work. Selection text is now
  capped at 20,000 characters before any processing.
- Settings' "learned style" section only populated once per app launch
  (`.onAppear` on a reused `NSWindow` doesn't reliably re-fire — the exact
  trap `openDraftsWindow` already worked around for its own counts).
  `styleProfiles` moved to `DraftEngine` as a `@Published` property,
  refreshed on every Settings open exactly like `openDraftsWindow` does.
- A "Reset learned style" landing *during* `StyleLearner.learn`'s
  multi-second-to-minutes LLM merge calls could have its effect silently
  undone by that in-flight call writing back the pre-reset profile text.
  `DraftStore.styleProfilesEpoch()` (bumped by every reset) lets `learn`
  detect and discard a now-stale write instead.

**Verified:** `swift build`/`swift test` (439 tests, all passing — 12 new
since the review), and — in a later session — the real provider's end-to-end
Insert behavior against a live draft in Mail.app (see `docs/hardening.md`'s
Draft-Modus verification playbook), since scripted seeding of the production
`drafts.db`/`askmail.db` for an automated live-Mac test was correctly
declined by this session's safety tooling.

### Phase 4 follow-up — on-demand generation, Ollama auto-start, newsletter gate

A later session, prompted by two live-Mac gaps the manual pass above
surfaced, extended Phase 4 rather than opening a new phase:

- **Insert now keeps the quoted conversation.** It originally replaced the
  selection outright; it now prepends the draft above the kept quote (two
  blank lines between), matching how a human reply is actually composed.
- **Local Ollama auto-start.** `OllamaEngine.ensureRunning()` — installed but
  not running (`.stopped`) starts it, already-running or not-installed are a
  no-op — is now called at app launch, at the top of every Draft-Modus tick,
  and from both Services verbs. Previously a daemon quit independently of
  AskMail (e.g. via its own menu-bar "Quit") silently broke every subsequent
  tick with unlogged `Connection refused` errors until the user noticed and
  restarted it manually from Settings.
- **`NewsletterClassifier.isNewsletterAddress`**, a new hard gate alongside
  `isNoReplySender`: any sender address containing "newsletter" *anywhere*
  (not just as a local-part prefix, which `matchesNewsletterLocalPart`
  already weakly signals on) skips drafting outright. Closes a real gap a
  live thread exposed — `bafinDE@newsletter.gsb.bund.de` got a `ready` draft
  because the existing check only looked at the local part (`bafinDE`),
  never the domain.
- **Manual-trigger rule bypass.** Both Services verbs now bypass every
  auto-draft eligibility rule (the master `draftModeEnabled` toggle, the
  sender exclusion list, newsletter/no-reply classification) when explicitly
  invoked — those rules exist to gate *unattended* background drafting; a
  user who just clicked Insert/Regenerate has already supplied the consent
  they exist to approximate. Insert, when no `ready` draft exists yet
  (never enqueued, or previously skipped by one of those rules), now
  generates on demand instead of just erroring; `SQLiteStore.
  latestThreadID(fromSenderAddress:)` resolves the target thread directly
  from the mailbox for this case, since a skipped/never-enqueued thread has
  no `drafts.db` row to match against at all.
- **Feedback without Automation.** A classic Services call can't hold open
  for an LLM call, and can't push text into Mail's compose window after
  returning without Phase 5's Automation grant — so on-demand generation
  sets the Services error-bezel text immediately ("Drafting a response
  now…", the only synchronous UI channel available) and posts a system
  notification when generation completes. True zero-click "generates and
  lands in the already-open compose window" needs Phase 5.

### Phase 6 — Settings UI (pending merge)

The already-shipped pieces (master toggle, battery policy, job-queue status
line) needed no new work — confirmed still correct, no changes. What was
genuinely new, per the plan's own scoping:

- **Sender exclusion list**: `SettingsStore.draftExcludedSenders` (`[String]`,
  each entry a bare address or domain) + a minimal list editor in Settings
  (no prior list-editor pattern existed to reuse, confirmed a third time).
  `SenderExclusion.isExcluded` (`Sources/AskMailCore`, unit-tested) is wired
  into `classifyPendingJobs` as a new hard gate alongside
  `isAutoGenerated`/`isNoReplySender`, reusing the `newsletterSkipped` state
  for the same reason `isNoReplySender` does — the outcome is identical.
- **Learned-style transparency**: `DraftStore.allStyleProfiles()` backs a new
  Settings section showing each scope's `sample_count`/`updated_at`
  ("learned from N replies, updated <date>"), extending the same
  transparency principle Q&A citations already give the user to "what shaped
  this draft's tone."
- **Reset learned style**: `DraftStore.deleteStyleProfiles(scope:)` (nil
  deletes every scope) + a destructive-confirmation Settings action.
  Deliberately does **not** reset `drafts.style_learned_at` markers — an
  already-examined draft's evidence is gone, so it correctly stays
  unexaminable; the profile stays empty until genuinely new Sent replies are
  learned from, matching the phase's own DoD.
- **`docs/hardening.md` DM section**: H-24/H-25/H-26/H-27, a dedicated
  Draft-Modus hardening posture section, and a Draft-Modus-specific
  verification playbook appended to the existing one.
- Permission-prompt UI for Phase 5 is correctly still absent — Phase 5
  hasn't shipped.

### Process note (recommend continuing for Phases 4–6)

Phases 2 and 3 were both built the same way: a research pass grounding the
design in the real codebase, direct implementation with continuous
build/test iteration, then a multi-dimension adversarial review (e.g.
concurrency, hardening/privacy, state-machine edge cases, integration with
prior phases) with every finding independently re-verified before being
trusted, then fixes. This caught real, non-obvious bugs both times (a
`MailboxWatcher` data race confirmed via ThreadSanitizer in Phase 2; a
system-vs-user prompt-channel gap and a sibling-draft double-counting bug in
Phase 3) that a single implementation-then-test pass did not surface.
Recommended for Phases 4–6 too, scaled to what each phase actually touches.

## Phase 5 — Global hotkey + Mail.app Automation

**Goal:** a hotkey-driven path to insert/regenerate without going through the
Services menu — AskMail's first-ever Automation grant.

**Blocked on interactive spikes on a live Mac before any design is finalized
here** (this is why this phase stays a stub, not a detailed plan, until
those run):
1. `sdef /Applications/Mail.app` — verify the actual AppleScript object model
   (message/mailbox/document hierarchy) on the target macOS version; prior
   phases' own experience (`Defaults.mailRoot`'s `V<n>` version-probing, the
   Accounts4.sqlite discovery in spike B11 #1) shows Apple's own on-disk/
   scripting surfaces drift across releases and must be verified live, never
   assumed from documentation.
2. Whether inserting text into Mail's compose window via Automation requires
   an Accessibility grant (synthetic keystroke/paste) or can go through a
   pure AppleScript/Apple Events object-model write. `docs/hardening.md`'s
   existing regression guard is explicit: **no `CGEventTap`, no Accessibility
   grant** ("Global hotkey via Carbon `RegisterEventHotKey`" — see
   `HotkeyManager.swift:40` — "no keystroke-interception surface"). If
   Automation-based insertion turns out to require Accessibility trust, this
   phase ships **clipboard-only** (copy the draft, hotkey surfaces a
   copy-and-notify affordance, user pastes manually) — never a transitional
   escalation to Accessibility as a stopgap.
- Once the spikes resolve those two questions, this section should be
  rewritten with the same level of detail as Phase 1's original plan (exact
  new types, exact `MailQuery.swift` surface, exact hardening-doc additions)
  before implementation starts.
- **DoD (provisional, pending the spikes):** the hotkey triggers either a
  live insertion into Mail's compose window or a clipboard-copy-and-notify
  fallback, whichever the spikes determine is safe; `docs/hardening.md`
  gains a new item documenting the Automation grant's scope and the
  Accessibility decision, with the same DoD rigor as existing H- items.
- **Permission-prompt UI** (Settings, once this phase ships): its Automation
  (and possibly Accessibility) grant needs an onboarding/status affordance,
  following `SettingsView.swift`'s actual `openFullDiskAccessSettings`-style
  pattern (a real Settings-pane deep link + status check), **not**
  `OnboardingChecklist.swift`, which is a plain data struct with no UI of
  its own. Not yet built — Phase 6 shipped without it since this phase
  hadn't landed yet (see Phase 6's entry under "What's shipped").
